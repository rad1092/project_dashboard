# 페이지와 기능 확장 가이드

## 새 페이지가 필요한 경우

- 현재 동작과 미래 확장을 분리해야 할 때
- 추천 결과와 분석 결과처럼 읽는 목적이 분명히 다를 때
- 설명용 화면이 별도로 필요할 때

즉, "한 파일이 너무 길어졌다"보다 "읽는 목적이 달라졌다"가 새 페이지를 만드는 더 좋은 기준이다.

## 기존 페이지 확장으로 충분한 경우

- 추천 카드에 정보 한두 개를 더 넣는 수준
- 분석 필터와 차트를 조금 추가하는 수준
- About 문구나 Projects 카드 업데이트

이 경우에는 새 페이지를 만들기보다 기존 페이지와 컴포넌트의 책임을 유지하는 쪽이 더 단순하다.

## 현재 구조에서 확장 우선순위

1. `2 대피소 추천` 고도화
2. `4 실시간 준비` 를 실제 실시간 페이지로 전환
3. `6 Data Analysis` 에 추가 분석 지표 반영
4. 필요 시 관리자용 또는 데이터 품질 점검용 새 페이지 추가

이 우선순위는 "핵심 사용자 흐름 -> 미래 확장 준비 -> 보조 분석" 순서를 반영한다.

## 새 기능을 넣을 때 확인할 것

- 지금 기능이 현재 동작인지 미래 준비 기능인지 구분했는가?
- 서비스 계층을 재사용할 수 있는가?
- docs 와 테스트까지 함께 바꿀 범위인가?
- `감지 지역`, `활성 지역`, `전용/기본/대체 대피소`, `직선 거리` 같은 핵심 용어 의미가 흔들리지 않는가?

## 실무적으로 보면 좋은 기준

- 입력 방식이 바뀌면 먼저 `2 대피소 추천` 과 데이터 서비스를 본다.
- 설명 방식이 바뀌면 `3 작동 설명`, `content.py`, 관련 docs 를 같이 본다.
- 실시간 기능이 붙으면 `4 실시간 준비` 내용을 실제 동작 기준으로 재작성한다.
- 분석 지표가 늘어나면 `6 Data Analysis` 와 `dashboard/components/charts.py` 를 함께 본다.
