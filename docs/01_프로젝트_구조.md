# 프로젝트 구조

## 현재 실행 파일

- `app.py`
  - 홈 화면만 담당한다.
  - 앱 소개, KPI, 데이터 상태 요약, 페이지 이동 안내를 보여준다.
- `dashboard_data.py`
  - 여러 화면이 같이 쓰는 데이터 로딩 기준을 모아 둔 공용 모듈이다.
  - `preprocessing_data` 경로를 찾고, CSV를 읽고, 공통 컬럼 형태로 정리한다.
- `pages/1_대피소_추천.py`
  - 사용자가 입력한 좌표를 기준으로 가장 가까운 지역을 추정하고, 재난 유형에 맞는 대피소 후보를 계산한다.
- `pages/2_실시간_테스트.py`
  - 브라우저 위치와 도보 경로 조회를 붙여서 추천 흐름을 시험하는 화면이다.
- `pages/3_Data_Analysis.py`
  - 과거 특보 이력과 대피소 분포를 차트로 읽는 화면이다.

## 왜 `dashboard_data.py`가 따로 있는가

예전에는 홈, 추천, 분석 화면이 비슷한 CSV 로더와 전처리 코드를 각각 들고 있었다. 그러면 아래 문제가 생긴다.

- 한 화면만 컬럼 이름 처리 기준이 달라질 수 있다.
- 데이터 경로 우선순위를 바꾸면 여러 파일을 같이 수정해야 한다.
- `app.py`가 홈 화면 파일인데도 데이터 처리 코드까지 많이 안게 된다.

그래서 지금은 데이터 관련 기준을 `dashboard_data.py` 한 곳으로 모아 두고, 화면 파일은 그 결과를 가져다 쓰는 구조로 정리했다.

## 폴더 기준으로 보면

```text
.
|- app.py
|- dashboard_data.py
|- pages/
|  |- 1_대피소_추천.py
|  |- 2_실시간_테스트.py
|  `- 3_Data_Analysis.py
|- preprocessing_data/
|- docs/
`- tests/
```

## 읽는 순서 추천

1. `app.py`
2. `dashboard_data.py`
3. `pages/1_대피소_추천.py`
4. `pages/2_실시간_테스트.py`
5. `pages/3_Data_Analysis.py`

이 순서로 보면 "홈 -> 공통 데이터 기준 -> 추천 -> 실시간 확장 -> 분석" 흐름이 잡힌다.
