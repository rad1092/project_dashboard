# 프로젝트 구조

## 현재 실행 파일

- `app.py`
  - 홈 화면만 담당한다.
  - 앱 개요, KPI, 데이터 상태 요약, 페이지 이동 안내를 보여준다.
- `dashboard_data.py`
  - 여러 화면이 같이 쓰는 기존 전처리 CSV 로딩 기준을 모아 둔 공용 모듈이다.
  - `preprocessing_data` 경로를 찾고, CSV를 읽고, 공통 컬럼 형태로 정리한다.
- `pages/1_대피소_추천.py`
  - 사용자 좌표를 기준으로 가까운 지역을 추정하고, 정적 CSV 기준 대피소 추천 로직을 파일 내부에서 직접 계산한다.
- `pages/2_실시간_테스트.py`
  - 브라우저 위치, OSRM 경로, 추천 계산까지 모두 파일 내부에서 독립적으로 처리하는 페이지다.
- `pages/3_재난문자_대피_안내.py`
  - 크롤링된 최근 재난문자를 현재 위치와 연결해 대피소 안내를 만드는 페이지다.
- `pages/4_Data_Analysis.py`
  - 과거 특보 이력과 대피소 분포를 차트로 보는 페이지다.

## 왜 `dashboard_data.py`가 따로 있는가

예전에는 홈, 추천, 분석 화면이 비슷한 CSV 로더와 전처리 코드를 각각 들고 있었다. 그러면 아래 문제가 생긴다.

- 한 화면만 컬럼 이름 처리 기준이 달라질 수 있다.
- 데이터 경로 우선순위를 바꾸면 여러 파일을 같이 수정해야 한다.
- `app.py`가 홈 화면 파일인데도 데이터 처리 코드까지 많아진다.

그래서 홈과 분석이 쓰는 기존 전처리 CSV 기준만 `dashboard_data.py`에 남기고, 추천/실시간/재난문자 페이지는 각자 필요한 로딩과 계산을 파일 안에서 직접 들고 있는 구조로 정리했다.
