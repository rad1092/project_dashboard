# 코드와 클래스 설계 가이드

## 이 문서의 목적

이 문서는 현재 코드만 설명하는 문서가 아니다. 앞으로 새 함수, 새 클래스, 새 서비스, 새 API 연동을 추가할 때 무엇을 조심해야 하는지 정리하는 설계 원칙 문서다.

## 함수와 클래스를 나누는 기준

### 함수로 충분한 경우

- 입력이 명확하고 출력이 하나로 정리되는 경우
- 상태를 오래 들고 있을 필요가 없는 경우
- 단순 변환, 집계, 포맷팅, 차트 생성 같은 작업

현재 저장소의 대부분은 이 경우라서 함수 중심 구조가 맞다.

### 클래스가 필요한 경우

- 상태를 여러 메서드가 공유해야 하는 경우
- 설정값과 동작을 묶는 것이 더 자연스러운 경우
- 외부 서비스 클라이언트처럼 연결 정보와 동작을 같이 들고 있어야 하는 경우

### 클래스를 만들 때 주의할 점

- 한 클래스가 너무 많은 책임을 갖지 않게 한다.
- UI, 데이터 가공, 외부 요청을 한 클래스에 같이 넣지 않는다.
- 테스트 가능한 작은 단위로 유지한다.

## API 호출 설계 원칙

- 페이지 파일에서 직접 API를 호출하지 않는다.
- 외부 요청은 `services` 계층에서 처리한다.
- 응답 정규화와 KPI 계산을 섞지 않는다.
- timeout, retry, cache, rate limit을 함께 고려한다.

## 반복 호출과 루프의 문제

예를 들어 목록 API를 받고, 그 결과를 돌면서 상세 API를 하나씩 호출하면 아래 문제가 생긴다.

- 느리다.
- API 사용량이 급격히 늘어난다.
- rate limit에 쉽게 걸린다.
- 실패 지점이 많아진다.
- Streamlit rerun 때 같은 요청이 반복될 수 있다.

즉, 제한 횟수를 고려하지 않고 루프문으로 API를 계속 불러오는 방식은 피해야 한다.

## 권장 패턴

- 가능하면 한 번에 필요한 데이터를 주는 API를 찾는다.
- 목록과 상세를 꼭 나눠야 한다면 캐시와 호출 간격을 고려한다.
- 서비스 함수 하나가 "요청 → 정규화 → 예외 처리"를 담당하게 만든다.
- 페이지는 결과를 받아서 보여주는 역할에 집중한다.

## 현재 저장소 기준 예시

### 좋은 방향

- `analysis_data.py`가 데이터를 준비한다.
- `pages/3_Data_Analysis.py`는 필터와 출력에 집중한다.

### 피해야 할 방향

- `pages/3_Data_Analysis.py` 안에서 `requests.get()`를 여러 번 직접 호출하는 것
- 차트 함수 안에서 외부 요청까지 같이 처리하는 것

## 클래스 설계 체크 포인트

- 이 상태를 정말 객체로 오래 들고 있어야 하는가?
- 함수 몇 개로 더 단순하게 표현할 수 없는가?
- 테스트할 때 외부 의존성을 쉽게 분리할 수 있는가?
- 이 클래스가 UI와 데이터 계층을 동시에 건드리지 않는가?

## 잘못 설계하면 생길 문제

- 클래스가 비대해져 이해와 테스트가 어려워진다.
- 외부 API나 DB 오류가 UI 계층까지 바로 전파된다.
- 한 곳을 고치려면 여러 책임을 같이 건드려야 한다.
