# 코드와 클래스 설계 가이드

## 기본 원칙

- 페이지는 화면 조합에 집중한다.
- CSV 로딩, 정규화, 추천 규칙은 서비스 계층에 둔다.
- 반복 UI 는 컴포넌트로 분리한다.
- 포맷 문자열은 유틸리티로 분리한다.

이 원칙을 지키는 이유는,
초보자가 파일을 열었을 때도 "이 파일이 무슨 책임을 가지는지"를 빠르게 알 수 있게 하기 위해서다.

## 함수로 충분한 경우

- 거리 계산
- 재난 정규화
- 특보 요약
- 차트 생성
- 지도 렌더링

현재 저장소는 대부분 함수 중심 구조가 적합하다.
상태를 오래 들고 있지 않는 계산이 많기 때문에, 작은 함수 여러 개로 나누는 편이 읽기 쉽다.

## 클래스를 만들 때

현재는 `DisasterDatasetBundle` 처럼 데이터 묶음을 표현하는 작은 구조체 수준만 사용한다.
상태를 오래 들고 있는 큰 클래스를 만들 필요는 아직 없다.

즉, "계산 로직을 담는 큰 서비스 클래스"보다
"데이터 묶음 객체 + 여러 개의 순수 함수" 조합이 현재 단계에 더 잘 맞는다.

## 금지에 가까운 패턴

- 페이지 파일 안에서 직접 `requests.get()` 반복 호출
- 외부 CSV 경로를 절대경로 문자열로 하드코딩
- 추천 규칙을 여러 페이지에 중복 작성
- 유료 API 의존 코드를 현재 단계에 바로 넣는 것
- `감지 지역` 과 `활성 지역` 개념을 섞어 쓰는 것

## 주석 작성 원칙

- 모듈 상단에서는 파일의 역할과 수정 진입점을 설명한다.
- 블록 주석에서는 화면 흐름, 데이터 흐름, 현재/미래 경계를 설명한다.
- 인라인 주석은 fallback, `session_state`, `copy()`, `groupby()`, 정렬 기준처럼 초보자가 막힐 지점에만 둔다.
- 같은 파일 안에서는 `첫 등장만 자세히` 설명하고, 반복되는 같은 패턴은 다시 길게 풀지 않는다.
- 코드 자체 설명이 필요할 때도 문법 번역보다 "이 줄이 실행되면 어떤 결과가 생기는지"를 한 문장으로 적는다.
- 변수명만 봐도 분명한 단순 대입에는 주석을 과도하게 늘리지 않는다.
