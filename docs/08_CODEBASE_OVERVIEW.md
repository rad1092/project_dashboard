# 코드베이스 개요

## 전체 흐름

1. `app.py` 가 홈 화면을 연다.
2. `dashboard/services/disaster_data.py` 가 외부 전처리 폴더를 찾고 CSV 를 읽는다.
3. `dashboard/services/shelter_recommendation.py` 가 재난 그룹과 추천 규칙을 적용한다.
4. `pages/2_대피소_추천.py` 가 결과를 지도와 카드로 보여준다.
5. `pages/3_작동_설명.py` 가 같은 흐름을 설명한다.
6. `pages/4_실시간_준비.py` 가 미래 확장 위치를 보여 준다.
7. `pages/6_Data_Analysis.py` 가 과거 특보와 대피소 분포를 분석한다.

## 데이터 흐름

- 입력: 위도, 경도, 재난 유형
- 중간 해석: 통합 대피소 평균 좌표를 기준으로 감지 지역을 만들고, 필요하면 수동 보정 지역을 활성 지역으로 사용
- 특보 데이터: `danger_clean.csv`
- 대피소 데이터: `final_shelter_dataset.csv`, `earthquake_shelter_clean_2.csv`, `tsunami_shelter_clean_2.csv`
- 처리: 재난 유형 정규화 → 후보 대피소 선택 → 거리 계산 → Top 3 정렬
- 출력: 추천 카드, 지도, 최근 특보 표

현재 구조가 이렇게 바뀐 이유는 사용자 입장에서 "내 위치"가 먼저 기준이 되기 때문이다.
다만 지금은 행정경계 기반 역지오코딩이 없어서, 지역 자동 감지는 통합 대피소의 지역 중심 좌표와의 근접 비교로만 동작한다.
그래서 추천 페이지에는 감지 결과를 사람이 보정할 수 있는 수동 지역 수정 UI 를 남겨 두었다.

## 왜 이렇게 나눴는가

- 페이지는 화면 조합에 집중한다.
- 서비스는 데이터 계약과 계산을 담당한다.
- 컴포넌트는 반복 UI 와 지도/차트 렌더링을 맡는다.
- 문서는 현재 구조와 미래 확장을 설명한다.

이 분리를 유지해야 실시간 API 가 들어와도 전체 구조가 무너지지 않는다.
