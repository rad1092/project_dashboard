# 데이터 소스 확장 가이드

## 이 문서의 목적

현재 저장소는 `load_demo_dataset()` 기반 샘플 데이터를 사용한다. 이 문서는 그 구조를 실제 CSV, API, DB로 확장할 때 어떤 순서와 기준으로 진행해야 하는지 설명한다.

## 기본 원칙

- 페이지는 데이터 소스를 몰라도 되게 만든다.
- 데이터 정규화는 서비스 계층에서 끝낸다.
- 반환 컬럼 구조를 최대한 안정적으로 유지한다.
- 외부 데이터 연결 전 호출량과 실패 처리를 먼저 설계한다.

## 1. 샘플 데이터에서 CSV로 확장

### 순서

1. CSV 파일 위치와 컬럼 정의 확인
2. `analysis_data.py`에 로딩 함수 추가
3. 컬럼명을 현재 구조에 맞게 정규화
4. KPI와 차트 함수가 그대로 작동하는지 확인
5. 테스트 갱신

### 주의점

- CSV 경로를 절대경로로 두지 않는다.
- 파일 형식 예외와 빈 파일 상황을 고려한다.

## 2. CSV에서 API로 확장

### 순서

1. 인증 방식과 호출 제한 확인
2. 요청 함수 작성
3. 응답을 DataFrame으로 변환
4. 누락 컬럼 보완 또는 정규화
5. 캐시와 예외 처리 추가
6. 페이지 연결

### 주의점

- rate limit과 pagination을 먼저 파악한다.
- 실패 응답을 그냥 비어 있는 데이터처럼 처리하지 않는다.
- 페이지에서 직접 API를 호출하지 않는다.

## 3. API에서 DB로 확장

### 순서

1. 저장할 데이터와 읽을 데이터를 구분
2. 모델과 연결 방식 정의
3. 읽기 함수와 쓰기 함수를 분리
4. 페이지에서는 기존 인터페이스를 최대한 유지

### 주의점

- 세션과 연결을 페이지 파일에서 직접 열지 않는다.
- 읽기 전용 조회와 쓰기 동작을 섞지 않는다.

## API 확장 시 특히 조심할 점

- `rate limit`: 분당, 시간당 제한을 먼저 확인
- `timeout`: 응답이 느릴 때 앱 전체가 멈추지 않게 설정
- `retry`: 실패 시 무한 재시도 금지
- `cache`: 같은 요청은 중복 호출하지 않게 함
- `N+1 요청`: 목록 후 상세를 루프로 반복 호출하는 패턴 금지
- `secrets`: 키와 토큰은 `.streamlit/secrets.toml`로 분리

## 예시 시나리오

### 나쁜 예시

- 프로젝트 50개 목록을 받고, 각 프로젝트 상세를 `for` 문 안에서 API로 다시 호출
- 사용량이 커지고 속도가 느려지고 실패 지점이 많아진다.

### 더 좋은 예시

- 가능하면 상세가 포함된 배치 응답을 사용
- 배치가 안 되면 상세 호출을 캐시하고 실패 시 제한적으로 재시도
- 서비스 계층에서 최종 DataFrame을 만든 뒤 페이지에 전달

## 체크할 테스트

- 컬럼 구조가 유지되는가
- 빈 응답 처리 로직이 있는가
- 실패 응답 메시지를 사용자가 이해할 수 있는가
- 기존 KPI 계산이 계속 맞는가
