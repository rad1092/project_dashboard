# Streamlit 학습 가이드

## 이 저장소로 배우는 핵심 포인트

- Streamlit 페이지를 기능별로 나누는 방법
- 페이지와 서비스 계층을 분리하는 방법
- 무료 지도와 차트를 조합해 설명 가능한 UI 를 만드는 방법
- 현재 구현과 미래 확장을 한 앱 안에서 섞지 않고 분리하는 방법

## 1. 홈과 멀티페이지 구조

이 저장소는 `app.py` 를 홈으로 두고 `pages/` 로 기능을 분리한다.
현재 구조에서는 소개, 추천, 설명, 실시간 준비, 분석을 각각 다른 페이지로 나눠 읽기 흐름을 만든다.

초보자에게 중요한 포인트는 Streamlit 이 한 파일짜리 데모 앱으로 끝나지 않는다는 점이다.
페이지 수가 늘어나도 홈과 사이드바 구조를 이용하면 역할별 탐색 흐름을 만들 수 있다.

## 2. 페이지와 서비스 분리

추천 규칙과 CSV 로딩을 페이지 안에서 직접 처리하지 않고 `dashboard/services` 로 분리했다.
이렇게 해야 나중에 실시간 API를 붙여도 페이지 코드는 크게 바뀌지 않는다.

특히 추천 페이지는 좌표 입력을 받은 뒤 지역 자동 감지까지 서비스 계층에서 처리한다.
이렇게 분리해 두면 페이지는 "어떤 값을 보여 줄지"에 집중하고,
서비스는 "좌표를 어떤 지역으로 해석할지" 같은 판단 로직을 맡는다.

코드를 읽을 때는 이렇게 생각하면 쉽다.

- 페이지: Streamlit 위젯, 레이아웃, 카드/표 배치
- 서비스: DataFrame 정리, 감지 지역 계산, 추천 후보 선정
- 컴포넌트: 반복 카드, 지도, 차트 렌더링

## 3. 무료 지도 렌더링

현재 지도는 `folium` 과 OSM 타일로 렌더링한다.
유료 지도 없이도 사용자 위치, 대피소 점, 직선 거리선을 충분히 보여줄 수 있다는 점을 학습 포인트로 삼는다.

같은 이유로 지역 자동 감지도 외부 reverse geocoding API 없이 로컬 데이터 기반으로 처리한다.
현재는 각 지역의 평균 좌표를 만든 뒤 가장 가까운 지역을 감지하는 방식이며,
이 한계를 화면과 문서 양쪽에서 함께 설명하는 것이 중요하다.

즉, 지금 지도는 "정확한 길찾기 도구"보다 "추천 근거를 시각적으로 이해시키는 도구"에 가깝다.

## 4. 설명 가능한 UI 만들기

`3 작동 설명` 페이지는 단순 결과 화면이 아니라, 추천이 어떻게 계산되는지 단계별로 설명한다.
이 구조는 Streamlit 이 단순 대시보드뿐 아니라 설명형 인터페이스에도 잘 맞는다는 점을 보여 준다.

초보자에게 특히 도움이 되는 부분은,
추천 결과와 해설 페이지를 분리해 두면 로직을 바꾸더라도 설명 기준을 같이 점검하게 된다는 점이다.

## 5. 미래 확장 분리

`4 실시간 준비` 페이지는 실제 기능을 넣지 않고도 미래 확장 지점을 문서화하는 예시다.
현재 구현과 미래 구현을 분리해 두는 습관 자체가 중요한 학습 포인트다.

나중에 reverse geocoding 또는 브라우저 geolocation 을 붙이게 되면
현재의 좌표 우선 흐름은 유지하되 `disaster_data.py` 의 지역 감지 부분만 교체하면 되도록 설계하는 것이 목표다.

정리하면, 이 저장소는 Streamlit 문법 자체보다도
"작은 앱을 계층과 문서까지 포함해 어떻게 구조화할 것인가"를 배우기 좋은 예제다.
