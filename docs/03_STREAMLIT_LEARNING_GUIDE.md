# Streamlit 학습 가이드


## 1. 페이지 설정과 멀티페이지 구조

### 무엇을 배우는가

- `st.set_page_config()`의 역할
- 루트 `app.py`와 `pages/` 구조
- 사이드바 기반 페이지 이동 방식

### 왜 필요한가

처음부터 페이지를 나누지 않으면 코드가 한 파일에 몰리고, 나중에 저장소 설명도 어렵다.

### 현재 코드에서 어디에 쓰였는가

- `dashboard/config.py`
- `app.py`
- `pages/1_About.py` ~ `pages/4_Learning_Log.py`

### 직접 수정 예시

- `dashboard/config.py`에서 홈 라벨을 바꾸고 홈 화면 문구가 어떻게 달라지는지 확인한다.
- `pages/`에 새 파일을 만들고 Streamlit 사이드바에 어떻게 나타나는지 확인한다.

### 주의점

- 페이지마다 설정을 중복으로 제각각 두면 앱 전체 톤이 흔들린다.
- 페이지 파일에 설정과 데이터 로직까지 몰아넣지 않는다.

## 2. 콘텐츠를 한 곳에서 관리하는 방법

### 무엇을 배우는가

- 설명 문구와 데이터 원본을 분리하는 이유
- `content.py` 같은 중앙 텍스트 파일의 장점

### 왜 필요한가

같은 문장이 여러 페이지에 흩어지면 수정할 때 누락이 생긴다.

### 현재 코드에서 어디에 쓰였는가

- `dashboard/content.py`
- `pages/1_About.py`
- `pages/2_Projects.py`
- `pages/4_Learning_Log.py`

### 직접 수정 예시

- `PROFILE_DATA["headline"]`를 바꿔보고 About 페이지에 반영되는지 확인한다.
- `PROJECT_ITEMS`의 카드 제목 하나를 바꿔 Projects 페이지에서 즉시 확인한다.

### 주의점

- 설명 문구를 페이지 파일에 하드코딩하면 나중에 README와 docs 설명이 어긋난다.
- 공통 문구를 옮길 때는 어떤 페이지가 그 값을 쓰는지 같이 확인한다.

## 3. 데이터 가공과 서비스 분리

### 무엇을 배우는가

- `DataFrame`을 만드는 로직과 화면 로직을 나누는 방식
- KPI 계산을 함수로 분리하는 이유

### 왜 필요한가

샘플 데이터에서 CSV, API, DB로 확장할 때 페이지 코드를 최소 수정으로 유지할 수 있기 때문이다.

### 현재 코드에서 어디에 쓰였는가

- `dashboard/services/analysis_data.py`
- `pages/3_Data_Analysis.py`

### 직접 수정 예시

- `load_demo_dataset()`의 컬럼 값을 하나 늘려보고 어떤 페이지와 테스트가 영향을 받는지 확인한다.
- `build_kpis()`에 새 지표를 추가한 뒤 `pages/3_Data_Analysis.py`에서 표시해본다.

### 주의점

- 페이지에서 직접 데이터 생성이나 외부 요청을 반복하면 구조가 금방 무너진다.
- API를 붙일 때는 루프 안에서 상세 요청을 여러 번 보내지 않도록 설계해야 한다.
- 캐시 없이 무거운 로딩 함수를 매번 실행하면 Streamlit rerun 때 체감 성능이 급격히 나빠진다.

## 4. 차트와 공통 UI 분리

### 무엇을 배우는가

- 차트 생성 함수를 분리하는 이유
- 공통 헤더와 카드 UI를 함수로 만드는 이유

### 왜 필요한가

같은 패턴이 여러 페이지에서 반복될 때, 한 곳만 고쳐도 전체 화면을 통일할 수 있기 때문이다.

### 현재 코드에서 어디에 쓰였는가

- `dashboard/components/layout.py`
- `dashboard/components/charts.py`
- `pages/1_About.py`
- `pages/3_Data_Analysis.py`

### 직접 수정 예시

- `build_status_share_chart()`의 제목을 바꿔서 페이지 반영을 본다.
- `render_page_intro()`를 수정해 여러 페이지의 헤더가 동시에 바뀌는지 확인한다.

### 주의점

- 차트 함수 안에 특정 페이지 전용 예외처리를 너무 많이 넣으면 재사용 가치가 떨어진다.
- 페이지 파일마다 비슷한 마크업을 복붙하지 않는다.

## 5. 포맷팅과 상태 라벨 처리

### 무엇을 배우는가

- 숫자, 퍼센트, 날짜, 상태명을 한 곳에서 바꾸는 방법
- 포맷 함수가 왜 작은 파일로 분리되는지

### 왜 필요한가

같은 데이터를 서로 다른 형식으로 보여주면 사용자가 혼란스럽다.

### 현재 코드에서 어디에 쓰였는가

- `dashboard/utils/formatters.py`
- `app.py`
- `pages/2_Projects.py`
- `pages/3_Data_Analysis.py`

### 직접 수정 예시

- `format_percent()` 자릿수를 바꾸고 분석 페이지 테이블이 어떻게 달라지는지 확인한다.
- `STATUS_LABELS`에 새 상태를 추가해본다.

### 주의점

- 페이지마다 각자 포맷팅하면 나중에 통일이 어렵다.
- 상태 라벨은 표시용과 내부 값의 역할을 섞지 않는 것이 좋다.

## 6. 테스트와 검증 습관

### 무엇을 배우는가

- 최소 테스트가 구조를 지켜주는 방식
- 문서 수정과 코드 수정을 같이 확인하는 습관

### 왜 필요한가

이 저장소는 공개용 설명 자료 성격도 있기 때문에, 작은 구조 오류도 신뢰도를 떨어뜨린다.

### 현재 코드에서 어디에 쓰였는가

- `tests/conftest.py`
- `tests/test_analysis_data.py`
- `tests/test_content.py`

### 직접 수정 예시

- `REQUIRED_COLUMNS`를 바꾼 뒤 어떤 테스트가 깨지는지 확인한다.
- `PROJECT_ITEMS`의 필수 키를 일부러 하나 빼고 테스트가 그 오류를 잡는지 확인한다.

### 주의점

- 테스트를 통과시켰다고 문서가 자동으로 맞아지는 것은 아니다.
- 반대로 문서만 수정하고 스모크 테스트를 생략하면 실행 단계에서 어긋남이 남는다.
