# 내보내기 가이드

## 이 문서의 목적

현재 저장소의 공식 산출물은 Markdown 문서다. 이 문서는 그것을 Word, PDF, PPT 형태로 옮길 때 어떤 기준으로 묶고 정리하면 되는지 설명한다.

## 왜 Markdown을 기준으로 두는가

- Git에서 변경 추적이 쉽다.
- 코드와 문서를 같은 저장소에서 같이 관리하기 좋다.
- Word/PDF/PPT는 최종 전달 형식이고, Markdown은 작업 원본이기 때문이다.

## Word로 옮길 때

### 추천 구성

1. 표지
2. 저장소 목적
3. 구조 설명
4. 수정 및 확장 절차
5. 코드 해설 요약
6. 개발 원칙과 주의사항
7. 체크리스트

### 합칠 문서 순서

- `README.md`
- `docs/01_PROJECT_DIRECTION.md`
- `docs/02_STRUCTURE_GUIDE.md`
- `docs/04_UPDATE_AND_EXPANSION_GUIDE.md`
- `docs/08`~`docs/12`
- `docs/14`, `docs/15`, `docs/16`

### 주의점

- Word는 길어질수록 목차를 먼저 잡아야 읽히기 쉽다.
- 코드 전체를 다 넣기보다 설명에 필요한 부분만 발췌한다.

## PDF로 옮길 때

### 추천 방식

- Word용 구조를 먼저 만든 뒤 PDF로 변환
- 또는 Markdown을 브라우저에서 열어 PDF 인쇄

### 주의점

- 코드 블록이 길면 페이지가 부자연스럽게 잘릴 수 있다.
- 표가 많으면 가로 폭을 먼저 점검해야 한다.

## PPT로 옮길 때

### 추천 슬라이드 구조

1. 저장소 목적
2. 왜 Streamlit 저장소로 정리했는가
3. 현재 구조 개요
4. 데이터 흐름 예시
5. 수정 절차와 확장 절차
6. 코드 설계 원칙
7. 앞으로의 확장 계획

### PPT에서 중요한 것

- 모든 문서를 슬라이드에 다 옮기지 않는다.
- 각 슬라이드는 핵심 주장 한 개만 담는다.
- 구조 설명은 트리나 흐름도 위주로 바꾼다.

## 수동 변환 체크리스트

- 제목과 문서 순서가 목적에 맞는가
- 코드 블록이 너무 길지 않은가
- README 문장과 본문 문장이 같은 메시지를 말하는가
- 최신 문서 기준으로 옮겼는가
