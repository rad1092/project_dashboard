# 라이브러리 가이드

## 이 문서의 목적

이 문서는 단순히 설치된 패키지 목록을 읽는 문서가 아니다. 각 라이브러리가 왜 들어가 있는지, 지금 실제로 쓰는지, 언제 써야 하는지, 어떤 점을 조심해야 하는지 정리하는 문서다.

## 현재 바로 쓰는 핵심 라이브러리

### `streamlit`

- 왜 들어가 있는가: 앱 화면과 페이지 구조를 만들기 위해
- 현재 실제 사용 여부: 사용 중
- 어디에 쓰는가: `app.py`, `pages/*`, `dashboard/config.py`, `dashboard/components/layout.py`
- 언제 더 쓰는가: 페이지, 위젯, 상태관리, 캐시, 파일 업로드를 추가할 때
- 주의점: rerun 특성을 이해하지 못하고 무거운 연산을 페이지에 직접 넣으면 체감 성능이 나빠진다.

### `pandas`

- 왜 들어가 있는가: 표 데이터 구성, 집계, 정렬, 컬럼 변환을 위해
- 현재 실제 사용 여부: 사용 중
- 어디에 쓰는가: `dashboard/services/analysis_data.py`, `dashboard/components/charts.py`, `dashboard/utils/formatters.py`
- 언제 더 쓰는가: CSV/API/DB 데이터를 DataFrame으로 정규화할 때
- 주의점: 페이지 파일에서 즉흥적으로 DataFrame 가공을 많이 하면 책임 분리가 무너진다.

### `numpy`

- 왜 들어가 있는가: 샘플 데이터 생성과 수치 계산을 위해
- 현재 실제 사용 여부: 사용 중
- 어디에 쓰는가: `dashboard/services/analysis_data.py`
- 언제 더 쓰는가: 더미 데이터, 통계 계산, 간단한 수치 시뮬레이션이 필요할 때
- 주의점: 실제 데이터 연결 이후에도 더미 생성 로직이 남아 있으면 혼동이 생길 수 있다.

### `plotly`

- 왜 들어가 있는가: 인터랙티브 차트를 만들기 위해
- 현재 실제 사용 여부: 사용 중
- 어디에 쓰는가: `dashboard/components/charts.py`, `pages/3_Data_Analysis.py`
- 언제 더 쓰는가: 추가 차트, 호버 정보, 범례, 다중 축이 필요할 때
- 주의점: 페이지 안에 차트 코드를 직접 늘리기보다 공통 함수로 옮기는 것이 낫다.

## 가까운 확장에서 쓸 가능성이 높은 라이브러리

### `requests`

- 왜 들어가 있는가: 외부 API 호출을 위해
- 현재 실제 사용 여부: 아직 직접 사용 안 함
- 언제 쓰는가: 실시간 데이터, 외부 서비스 연동, 공개 API 분석
- 주의점: 타임아웃, 재시도, rate limit, secrets 분리를 반드시 같이 고려한다.

### `sqlalchemy`

- 왜 들어가 있는가: DB 연결과 모델링을 위해
- 현재 실제 사용 여부: 아직 직접 사용 안 함
- 언제 쓰는가: 저장형 데이터, 사용자 입력 저장, 로그 기록이 필요할 때
- 주의점: 세션과 연결 관리를 페이지에서 직접 처리하지 않는다.

### `openpyxl`

- 왜 들어가 있는가: 엑셀 파일 입출력을 위해
- 현재 실제 사용 여부: 아직 직접 사용 안 함
- 언제 쓰는가: 엑셀 업로드/다운로드가 필요할 때
- 주의점: 파일 포맷 예외와 시트 구조 차이를 고려해야 한다.

### `pillow`

- 왜 들어가 있는가: 이미지 처리와 썸네일 작업을 위해
- 현재 실제 사용 여부: 아직 직접 사용 안 함
- 언제 쓰는가: 프로젝트 이미지, 미리보기, 업로드 이미지 가공이 필요할 때
- 주의점: 큰 이미지 파일은 메모리 사용량을 빠르게 늘린다.

## 시각화 확장용 라이브러리

### `matplotlib`, `seaborn`, `altair`, `folium`, `koreanize-matplotlib`

- 왜 들어가 있는가: Plotly 외의 시각화 선택지를 확보하기 위해
- 현재 실제 사용 여부: 현재 메인 앱에서는 사용 안 함
- 언제 쓰는가:
  - `matplotlib`, `seaborn`: 정적 분석 그래프
  - `altair`: 선언형 차트
  - `folium`: 지도 시각화
  - `koreanize-matplotlib`: matplotlib 한글 처리
- 주의점: 시각화 라이브러리를 여러 개 섞으면 스타일과 유지보수 비용이 늘어난다. 기본은 Plotly로 통일하고, 필요가 분명할 때만 추가한다.

## 분석 실험용 라이브러리

### `statsmodels`

- 왜 들어가 있는가: 통계 모델링과 회귀 분석 실험을 위해
- 현재 실제 사용 여부: 사용 안 함
- 언제 쓰는가: 단순 시각화가 아니라 분석 모델 결과를 설명할 때
- 주의점: 모델링 코드는 더 길고 복잡해지므로 노트북 실험과 앱 코드를 적절히 나눠야 한다.

### `jupyter`

- 왜 들어가 있는가: 탐색적 분석 실험을 위해
- 현재 실제 사용 여부: 사용 안 함
- 언제 쓰는가: 앱에 넣기 전 데이터 탐색과 실험을 먼저 해볼 때
- 주의점: 노트북에서 끝난 실험을 그대로 제품 구조로 착각하지 않는다.

## 특수 목적 라이브러리

### `opencv-python`, `opencv-contrib-python`

- 왜 들어가 있는가: 이미지/영상 처리 확장 가능성을 열어두기 위해
- 현재 실제 사용 여부: 사용 안 함
- 언제 쓰는가: 영상 분석, 이미지 전처리, 컴퓨터 비전 실험이 필요할 때
- 주의점: 지금 저장소의 중심은 데이터 정리와 설명이다. 필요가 생기기 전까지는 앱 구조에 억지로 넣지 않는다.

## 테스트용 라이브러리

### `pytest`

- 왜 들어가 있는가: 데이터 구조와 콘텐츠 기본 형태가 깨지지 않았는지 확인하기 위해
- 현재 실제 사용 여부: 사용 중
- 어디에 쓰는가: `tests/test_analysis_data.py`, `tests/test_content.py`
- 언제 더 쓰는가: 새 기능, API 연결, 새 데이터 컬럼이 생길 때
- 주의점: 테스트를 너무 세세한 문장 표현에 묶으면 문구 수정 때마다 불필요하게 깨질 수 있다.
