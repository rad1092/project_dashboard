# 이 저장소의 방향

## 저장소 목적

이 저장소는 Streamlit으로 프로젝트 결과, 데이터 분석 과정, 학습 내용을 정리하고 설명하기 위한 작업 공간이다. 목적은 화려한 대시보드를 만드는 것 자체가 아니라, 내가 무엇을 했고 왜 그렇게 했고 앞으로 어떻게 확장할지를 한 저장소 안에서 일관되게 보여주는 것이다.

## 이 저장소가 포트폴리오에서 맡는 역할

- 작업 기록: 프로젝트 목표, 역할, 결과, 다음 액션을 남긴다.
- 설명 자료: README와 docs를 통해 작업 맥락을 설명한다.
- 구조 증빙: 코드를 어떻게 나누고 유지했는지 보여준다.
- 학습 증빙: 단순히 배웠다고 말하는 것이 아니라, 배운 내용을 실제 구조에 반영한 흔적을 남긴다.

## 왜 문서를 이렇게 나누는가

문서가 하나에 다 몰려 있으면 찾기 어렵고, 수정 기준과 설명 기준이 섞인다. 이 저장소의 문서는 역할별로 나뉜다.

- `01`: 저장소 목적과 운영 기준
- `02`: 파일과 폴더 설명
- `03`: 이 저장소를 통해 배우는 개념
- `04`: 수정과 확장 절차
- `05`: 라이브러리 설명
- `06`: 이 저장소로 무엇을 했는지 설명하는 법
- `07`: 빠른 최종 점검
- `08`~`12`: 현재 코드 해설
- `13`: 외부 문서 형식으로 옮기는 기준
- `14`: 코드와 클래스 설계 원칙
- `15`: 데이터 소스 확장
- `16`: 페이지와 기능 확장

## 왜 코드, 문서, 테스트를 같이 관리해야 하는가

이 저장소는 공개용 설명 자료 성격이 강하다. 따라서 화면만 맞고 문서가 틀리거나, 문서만 맞고 테스트가 깨져 있으면 저장소 전체 신뢰도가 떨어진다.

- 코드: 실제 동작과 구조를 보여준다.
- 문서: 왜 그렇게 만들었는지 설명한다.
- 테스트: 최소한의 구조가 망가지지 않았는지 확인한다.

셋 중 하나만 맞아도 충분하지 않다.

## 개발 원칙과 금지 사항

### 개발 원칙

- 페이지 파일은 화면 조합에 집중하고, 데이터 준비와 계산은 별도 모듈로 분리한다.
- 자주 바뀌는 설명 문구는 한 곳에서 관리한다.
- 새 기능을 넣을 때는 먼저 현재 구조를 어디까지 재사용할 수 있는지 본다.
- 문서를 나중에 정리하지 말고, 구조를 바꾸는 날 같이 갱신한다.

### 금지 또는 강한 주의가 필요한 것

- 페이지 파일 안에서 직접 외부 API를 반복 호출하는 것
- rate limit을 고려하지 않고 루프 안에서 상세 API를 N번 호출하는 것
- 비밀정보를 코드에 직접 적는 것
- 절대경로나 로컬 환경 전용 경로를 쓰는 것
- 하나의 함수나 클래스에 너무 많은 책임을 몰아넣는 것
- 테스트와 문서를 건너뛰고 화면만 맞췄다고 끝내는 것

예를 들어 API를 불러올 때 제한 횟수를 고려하지 않고 `for` 문 안에서 상세 요청을 계속 보내면 느리기도 하고 사용량도 급격히 커진다. 이 저장소에서는 그런 실수를 막기 위해 `services` 분리, 캐시, 문서화, 체크리스트를 같이 둔다.
