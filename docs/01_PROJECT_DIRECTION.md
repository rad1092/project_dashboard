# 이 프로젝트의 방향

## 이 문서를 읽으면 알 수 있는 것

- 이 저장소가 왜 만들어졌는가
- 지금 실제로 구현된 범위와 일부러 막아 둔 범위가 무엇인가
- 이후 확장을 할 때 무엇을 유지해야 하는가

## 목표

이 저장소의 목표는 전처리된 재난 특보 이력과 대피소 데이터를 이용해 다음 두 가지를 함께 보여주는 것이다.

- 사용자가 바로 써볼 수 있는 대피소 추천 흐름
- 그 추천이 어떤 데이터와 규칙으로 만들어졌는지 설명하는 구조

즉, 단순히 화면을 만드는 프로젝트가 아니라 추천 결과와 구현 근거를 같이 남기는 프로젝트다.
초보자 관점에서 보면 이 프로젝트는 "추천 앱"과 "추천 설명서"가 한 저장소 안에 같이 있는 구조라고 이해하면 쉽다.

## 현재 단계

- 과거 전처리 CSV 기반
- 좌표 우선 입력 + 지역 자동 감지 + 필요 시 수동 보정
- 무료 OSM 지도 사용
- 직선 거리 기준 추천
- 실시간 API와 경로 API는 미연결

현재 추천 페이지는 사용자가 위도와 경도를 먼저 넣으면,
통합 대피소 데이터에서 계산한 지역 중심 좌표 중 가장 가까운 곳을 기준으로 시도/시군구를 자동 감지한다.
즉, 행정경계 기반 위치 판별이 아니라 로컬 데이터 기반 근접 추정이며,
감지 결과가 애매할 때는 사용자가 직접 지역을 보정할 수 있게 둔다.

여기서 중요한 용어는 두 가지다.

- `감지 지역`: 좌표를 기준으로 자동 추정한 지역
- `활성 지역`: 감지 지역 또는 수동 보정 지역 중 실제 추천과 특보 요약에 쓰는 최종 지역

추천 화면의 나머지 계산은 모두 이 `활성 지역` 을 기준으로 움직인다.

## 핵심 원칙

- 유료 API와 유료 지도 API는 현재 단계 코드에 넣지 않는다.
- 외부 전처리 CSV는 읽기 전용으로 사용한다.
- 페이지는 화면 조합에 집중하고, 데이터 로딩과 추천 규칙은 서비스 계층으로 분리한다.
- 코드, 문서, 테스트는 같은 날 함께 갱신한다.

즉, "작동하는 코드"만 늘리는 것이 아니라 "왜 이렇게 작동하는지 설명하는 문서와 테스트"도 같이 관리하는 것이 원칙이다.

## 왜 페이지를 나눴는가

- `2 대피소 추천`: 실제 사용자 흐름
- `3 작동 설명`: 계산 근거와 데이터 흐름
- `4 실시간 준비`: 미래 확장 위치
- `6 Data Analysis`: 과거 이력 분석

이렇게 나누면 현재 동작과 미래 확장을 한 화면에 섞지 않아도 된다.
특히 `2 대피소 추천` 과 `3 작동 설명` 을 분리한 이유는,
한 화면에서는 사용자가 결과를 보고 다른 화면에서는 개발자와 리뷰어가 근거를 보게 하기 위해서다.

## 다음 확장 방향

1. 브라우저 위치 권한 연동
2. 실시간 특보 API 연동
3. 경로 안내 API 연동 여부 검토
4. 추천 정확도와 설명 문구 보강

좌표 우선 구조를 택한 이유는,
사용자가 이미 현재 위치를 알고 있을 때 지역을 먼저 다시 고르게 하는 흐름이 불필요하게 느껴지기 때문이다.
대신 현재 단계에서는 외부 역지오코딩 API 를 쓰지 않으므로,
자동 감지 결과가 언제든 수동 보정 가능해야 한다는 제약을 함께 문서화한다.

이 문장을 코드 쪽으로 옮기면 다음과 같이 해석할 수 있다.

- 좌표 입력 UI 를 먼저 둔다.
- 감지 지역 계산은 서비스 계층에서 한다.
- 감지 결과를 그대로 강제하지 않고 `지역 직접 수정` UI 를 둔다.
- 실시간 기능이 붙더라도 이 기본 흐름은 최대한 유지한다.
