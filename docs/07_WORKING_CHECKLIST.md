# 작업 체크리스트

## 이 문서의 용도

이 문서는 설명 문서가 아니라 빠른 최종 점검 문서다. `docs/04`가 수정 절차를 설명하고, `docs/14`가 설계 원칙을 설명한다면, 이 문서는 실제 작업 전과 커밋 전에 빠르게 확인하는 용도로 쓴다.

## 왜 체크해야 하는가

작은 저장소는 실수도 작게 보이기 쉽다. 하지만 이 저장소는 공개용 설명 자료 역할을 하므로, 아래 같은 실수가 남으면 품질이 바로 드러난다.

- 문서와 코드가 서로 다른 설명을 함
- 페이지 파일에 로직이 과도하게 몰림
- API 사용량과 속도를 고려하지 않은 구현
- 비밀정보 노출
- 테스트 누락

그래서 체크리스트는 "형식적인 습관"이 아니라 실수 방지 장치다.

## 언제 보는가

- 새 기능을 넣기 전
- README나 docs를 크게 수정한 뒤
- API/CSV/DB 연결 작업 직전
- 커밋 직전

## 작업 전 체크

- [ ] 이번 작업이 문구 수정인지, 구조 수정인지, 확장 작업인지 먼저 구분했는가?
- [ ] 먼저 볼 문서를 정했는가?
  - 구조: `docs/02_STRUCTURE_GUIDE.md`
  - 수정/확장: `docs/04_UPDATE_AND_EXPANSION_GUIDE.md`
  - 데이터 확장: `docs/15_DATA_SOURCE_EXPANSION_GUIDE.md`
  - 페이지 확장: `docs/16_PAGE_AND_FEATURE_EXPANSION_GUIDE.md`
- [ ] 새 폴더나 새 파일이 정말 필요한 수준의 변경인가?

## 코드 작성 중 체크

- [ ] 페이지 파일에서 처리할 일과 서비스 모듈에서 처리할 일을 구분했는가?
- [ ] 숫자/날짜 포맷을 여러 군데에서 중복 작성하지 않았는가?
- [ ] 하드코딩 문구를 `dashboard/content.py`로 옮길 필요는 없는가?
- [ ] 하나의 함수나 클래스가 너무 많은 책임을 가지지 않는가?

## API와 데이터 체크

- [ ] API 호출을 페이지 파일에서 직접 처리하지 않았는가?
- [ ] rate limit과 timeout을 고려했는가?
- [ ] 루프 안에서 상세 API를 여러 번 호출하는 구조가 아닌가?
- [ ] 캐시 없이 동일한 요청을 반복하고 있지 않은가?
- [ ] 샘플 데이터와 실제 데이터를 혼동하지 않도록 구분했는가?

## 보안과 설정 체크

- [ ] 비밀정보를 코드에 직접 넣지 않았는가?
- [ ] `.streamlit/secrets.toml`이 Git에 올라가지 않도록 했는가?
- [ ] 절대경로나 로컬 환경 전용 경로를 쓰지 않았는가?

## 문서 체크

- [ ] README의 실행 방법과 문서 목록이 현재 구조와 맞는가?
- [ ] 새 페이지나 새 폴더를 만들었다면 관련 docs도 같이 갱신했는가?
- [ ] 저장소 목적 설명이 실제 코드와 맞는가?

## 테스트와 확인 체크

- [ ] 기본 테스트가 통과하는가?
- [ ] `streamlit run app.py`로 주요 페이지가 열리는가?
- [ ] 깨진 문자열이나 인코딩 문제가 없는가?
- [ ] 변경 사항을 처음 보는 사람이 읽어도 흐름을 이해할 수 있는가?
