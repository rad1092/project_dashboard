# 이 저장소 설명 가이드

## 이 문서의 목적

이 문서는 포트폴리오 자체를 만드는 방법을 설명하는 문서가 아니다. 목적은 이 저장소를 바탕으로 내가 무엇을 했고, 왜 그렇게 했고, 어떻게 확장했는지를 외부에 설명할 때 쓸 문장 기준을 주는 것이다.

## 기본 설명 구조

이 저장소를 설명할 때는 아래 순서로 말하면 정리가 잘 된다.

1. 무엇을 정리하는 저장소인지
2. 왜 이 구조가 필요한지
3. 내가 어떤 역할을 했는지
4. 어떤 기준으로 확장하고 있는지
5. 앞으로 무엇을 붙일 예정인지

## README용 짧은 소개 예시

- "Streamlit을 이용해 프로젝트 결과, 데이터 분석 과정, 학습 내용을 정리하고 설명하는 저장소입니다."
- "코드, 문서, 테스트를 함께 관리해 작업 과정을 설명 가능한 형태로 남기는 저장소입니다."

## GitHub 소개 문장 예시

- "프로젝트 결과와 분석 과정을 Streamlit 기반 구조로 정리하고, 문서와 테스트까지 함께 유지하는 저장소"
- "단순한 화면 구현보다 구조 설명과 확장 가능성을 보여주기 위한 Streamlit 작업 공간"

## 자기소개서나 포트폴리오 서술 예시

- "산재한 실험 코드를 다시 정리해, 프로젝트 결과와 데이터 분석 과정을 설명 가능한 구조로 재구성했습니다."
- "페이지, 서비스, 문서, 테스트를 분리해 이후 실제 데이터와 기능을 연결하기 쉬운 형태로 정리했습니다."
- "단순히 대시보드를 만드는 것이 아니라, 왜 이런 구조로 나눴는지까지 설명할 수 있도록 문서 체계를 함께 구축했습니다."

## 면접 답변형 요약 예시

### 질문: 이 저장소에서 무엇을 했나요?

"Streamlit으로 화면을 만드는 것에 그치지 않고, 프로젝트 결과와 분석 과정을 정리할 수 있는 저장소 구조를 만들었습니다. 페이지는 화면 조합에 집중하게 하고, 데이터 준비와 문구 관리는 따로 분리해 이후 확장과 설명이 쉬운 형태로 구성했습니다."

### 질문: 왜 문서를 많이 만들었나요?

"이 저장소는 공개용 설명 자료 역할도 하기 때문입니다. README만으로는 부족한 부분을 구조 설명, 수정 절차, 코드 해설, 확장 가이드로 나눠서 처음 보는 사람도 흐름을 따라올 수 있게 했습니다."

### 질문: 앞으로 어떻게 확장할 건가요?

"현재는 샘플 데이터 기반 구조를 먼저 잡아두었고, 이후 CSV, API, DB로 데이터 소스를 확장하면서도 페이지 구조는 유지하도록 설계했습니다."

## 좋은 문장과 나쁜 문장

### 좋은 문장

- 무엇을 했는지 명확하다.
- 왜 그렇게 했는지가 들어간다.
- 구조나 판단이 드러난다.
- 이후 확장 방향이 보인다.

### 피해야 할 문장

- "열심히 만들었습니다."
- "다양한 기능을 넣었습니다."
- "좋은 경험이었습니다."

이런 문장은 실제 작업의 가치나 판단을 거의 보여주지 못한다.

## 설명할 때 자주 빠지는 것

- 왜 Streamlit을 썼는지
- 왜 페이지와 데이터 로직을 나눴는지
- 왜 문서와 테스트를 같이 관리하는지
- 왜 아직 샘플 데이터를 쓰고 있는지
- 실제 데이터 연결 시 무엇을 바꾸려는지

이 다섯 가지를 같이 말하면 설명의 밀도가 올라간다.
